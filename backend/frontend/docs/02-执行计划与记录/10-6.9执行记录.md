## 新增：后端启动容错改进（SQLite alter 表回退 + 端口占用友好提示）

### 问题
开发过程中 `sequelize.sync({ alter: true })` 在 SQLite 下会触发 `users_backup` 临时表的 `UNIQUE 约束冲突`，导致：
```
[DB] alter 同步失败 → 进程崩溃退出 → nodemon 反复重启失败
```

### 改进内容

| # | 文件 | 改动 |
|---|------|------|
| 1 | [`ziling-backend/src/app.js`](file:///d:/Ziling-app/backend/ziling-backend/src/app.js) | ✏️ `bootstrap()` 启动流程由 `await sequelize.sync({ alter: true })` 改为 **双层 try/catch**：先 alter，失败则降级为 `sync({ force: false })`（仅建缺失的表，不碰已有数据）；打印 "[DB] alter 同步失败，回退到安全模式..." 警告 |
| 2 | [`ziling-backend/src/app.js`](file:///d:/Ziling-app/backend/ziling-backend/src/app.js) | ✏️ `app.listen` 改为 `const server = app.listen(...)`，绑定 `server.on('error', ...)`：若为 `EADDRINUSE`（端口被占），打印友好提示 `❌ 端口 3000 已被占用，请先关闭占用程序或改 PORT`，而非默认的 Node 堆栈报错 |

### 启动流程（改进后）
```
Node 启动
  └─> sequelize.authenticate() → [DB] SQLite 连接成功
  └─> sequelize.sync({ alter: true })
         ├─ 成功 → [DB] 表结构同步完成
         └─ 失败 → [DB] alter 同步失败，回退到安全模式... → sync({ force: false })
  └─> 监听 3000 端口
         ├─ 成功 → 🚀 已启动 → http://localhost:3000
         └─ 失败（EADDRINUSE）→ ❌ 端口 3000 已被占用 → 请先关闭占用程序...
```

### 对用户测试的影响
- ✅ 首次使用全新数据库：正常 alter 同步，无任何变化
- ✅ 已有数据 + 本次发布无模型变更：sequelize 判定无需 alter，跳过
- ✅ 已有数据 + 后续版本有模型变更：alter 失败 → 自动降级为 force:false（仅创建缺失表）→ 正常启动，用户数据保留
- ✅ 端口被其他程序占用：不再打印晦涩堆栈，改为中文友好提示

### 备注
- `sync({ alter: true })` 在 SQLite 上的 alter 机制本质是「建 `xxx_backup` 表 → 删原表 → 建新表 → 从 backup 恢复」，与 MySQL/PostgreSQL 的原生 `ALTER TABLE` 不同，存在偶尔的约束冲突风险。双层 try/catch 消除了该风险对用户启动流程的阻塞。
- 端口占用检测仅处理 `EADDRINUSE`，其他错误仍打印堆栈便于排错。
