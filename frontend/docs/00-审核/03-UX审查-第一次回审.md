## 1️⃣ 日历页面 (page-prompt-calendar.md)

### 🔴 高优先级

#### 问题1：日期格子尺寸过小（40×40px），不符合44px触摸标准
- **位置**：组件尺寸表格，日期格子尺寸
- **问题**：设计规范要求"可点击元素最小44px"，但日历日期格子是40×40px，在移动端可能误触
- **影响**：用户可能点错日期，需要多次尝试才能选中正确日期
- **建议**：
  ```css
  /* 修正：日期格子至少 44×44px */
  --zl-calendar-day-size: 44px;  /* 从 40px 改为 44px */
  ```
- **修正成本**：低（仅修改设计规范 + tokens.css）

#### 问题2：非当月日期的点击行为未定义
- **位置**：日历网格功能说明
- **问题**：点击非当月日期（灰色淡化的日期）应该发生什么？是自动切换到那个月份并选中该日期，还是什么都不做？
- **影响**：用户可能尝试点击非当月日期，但没有反馈，会感到困惑
- **建议**：
  - 点击非当月日期 → 自动切换到那个月份，并选中该日期
  - 或者，点击非当月日期 → 什么都不做，但需要有一个提示（比如"点击 < 或 > 切换月份"）
- **修正成本**：低（仅添加交互逻辑）

#### 问题3：任务标记点更新时机未定义
- **位置**：日期小圆点显示规则
- **问题**：当一个任务被标记为完成后，日历网格中的任务标记点是否应该立即更新（消失或变灰）？还是在下次打开日历时才更新？
- **影响**：如果标记点不立即更新，用户可能认为任务还没有完成，导致重复操作
- **建议**：
  ```js
  // 在 toggleComplete() 后，立即更新日历网格
  function toggleComplete(taskId) {
    const task = tasks.value.find(t => t.id === taskId)
    if (task) {
      task.completed = !task.completed
      task.updatedAt = new Date().toISOString()
      saveToStorage(tasks.value)
      
      // 立即更新日历标记点
      updateCalendarMarks(task.date)
    }
  }
  ```
- **修正成本**：中（需要添加响应式更新逻辑）

### 🟠 中优先级

#### 问题4：月份切换缺少过渡动画
- **位置**：月份导航功能说明
- **问题**：提示词说"页面切换动画取消"，但月份切换时日历网格是否应该有一个淡出/淡入的过渡？突然刷新会让用户感觉突兀
- **影响**：用户体验感觉"硬"，不够流畅
- **建议**：
  ```css
  /* 月份切换过渡动画（150ms） */
  .calendar-grid {
    transition: opacity var(--zl-transition-fast);
  }
  
  .calendar-grid.switching {
    opacity: 0.5;
  }
  ```
- **修正成本**：低（仅添加 CSS 过渡）

#### 问题5：选中日期后页面不滚动，待办区可能在视口外
- **位置**：待办列表功能说明
- **问题**：在移动端，如果日历网格在视口上方，选中日期后待办区可能在视口外，用户可能不知道下面有内容
- **影响**：用户可能认为"选中日期后没有反应"，不知道向下滚动可以看到任务列表
- **建议**：
  ```js
  // 选中日期后，平滑滚动到待办区
  function selectDate(date) {
    selectedDate.value = date
    nextTick(() => {
      document.querySelector('.task-list').scrollIntoView({ 
        behavior: 'smooth',
        block: 'start'
      })
    })
  }
  ```
- **修正成本**：低（仅添加滚动逻辑）

#### 问题6：滑动手势的触发区域未定义
- **位置**：月份切换功能说明
- **问题**：滑动手势是在整个页面生效，还是仅在日历网格区域生效？如果整个页面都支持滑动，用户可能在滚动任务列表时误触发月份切换
- **影响**：用户在滚动任务列表时，可能误触发月份切换
- **建议**：
  - 滑动手势仅在日历网格区域生效
  - 或者，添加一个"滑动阈值"（比如滑动距离大于50px才触发）
- **修正成本**：低（仅添加手势区域判断）

#### 问题7：键盘导航不支持（无障碍访问）
- **位置**：整个文件
- **问题**：日历应该支持键盘导航（左右箭头切换日期，上下箭头切换周），但没有提到无障碍访问
- **影响**：键盘用户（比如使用屏幕阅读器的视障用户）无法使用日历
- **建议**：
  ```js
  // 键盘导航支持
  onMounted(() => {
    document.addEventListener('keydown', handleKeydown)
  })
  
  function handleKeydown(e) {
    if (e.key === 'ArrowLeft') {
      // 选中前一天
      selectPrevDay()
    } else if (e.key === 'ArrowRight') {
      // 选中后一天
      selectNextDay()
    } else if (e.key === 'ArrowUp') {
      // 选中上一周同一天
      selectPrevWeek()
    } else if (e.key === 'ArrowDown') {
      // 选中下一周同一天
      selectNextWeek()
    }
  }
  ```
- **修正成本**：中（需要添加键盘事件监听）

### 🟡 低优先级

#### 问题8：日期格式不统一
- **位置**：任务详情页结构
- **问题**：任务详情页显示"📅 21 Jun 2026"，但其他地方可能显示"2026-06-21"。格式应该统一
- **影响**：用户可能认为这是两个不同日期
- **建议**：
  - 统一使用"YYYY年MM月DD日"格式（中文环境）
  - 或者，统一使用"MMM DD, YYYY"格式（国际化）
- **修正成本**：低（仅修改显示格式）

#### 问题9：空状态文案太生硬
- **位置**：待办区空状态
- **问题**："今天没有任务"太生硬。是否应该更友好？
- **影响**：用户可能感觉"冷冰冰"，没有情感化设计
- **建议**：
  - "今天没有任务，要不要添加一个？"
  - 或者，"今天没有安排，享受自由时间吧！"
- **修正成本**：低（仅修改文案）

#### 问题10：返回按钮的行为未完全定义
- **位置**：任务详情页功能说明
- **问题**：从任务详情页返回日历时，之前的选中状态是否保留？滚动位置是否保留？
- **影响**：如果返回后滚动位置丢失，用户可能需要重新滚动到之前的位置
- **建议**：
  ```js
  // 使用 Vue Router 的 scrollBehavior
  const router = createRouter({
    history: createWebHistory(),
    scrollBehavior(to, from, savedPosition) {
      if (savedPosition) {
        return savedPosition
      } else {
        return { top: 0 }
      }
    }
  })
  ```
- **修正成本**：低（仅添加路由配置）

#### 问题2：表单验证错误提示样式未定义
- **位置**：新建任务弹窗
- **问题**：如果用户输入无效数据（比如结束时间早于开始时间），是否有错误提示？错误提示的样式是什么？
- **影响**：用户可能不知道为什么无法提交，需要猜测错误原因
- **建议**：
  ```vue
  <template>
    <div class="form-field" :class="{ 'has-error': errors.startTime }">
      <span class="field-icon">🕐</span>
      <input 
        type="time" 
        v-model="form.startTime" 
        class="field-input"
      />
      <span v-if="errors.startTime" class="error-message">
        {{ errors.startTime }}
      </span>
    </div>
  </template>
  
  <style scoped>
  .form-field.has-error .field-input {
    border-color: var(--zl-danger, #FF5252);
  }
  
  .error-message {
    font-size: var(--zl-font-xs);
    color: var(--zl-danger, #FF5252);
    margin-top: var(--zl-space-1);
  }
  </style>
  ```
- **修正成本**：中（需要添加验证逻辑 + 错误提示 UI）

#### 问题1：输入框和发送按钮的固定定位层级可能错误

- **问题**：提示词说"输入区的z-index要正确：输入区 fixed 定位要在 TabBar 上方（z-index > TabBar的z-index），但低于开屏层(z-index: 9999)"，但没有给出具体的 z-index 数值

  ```css
  /* 明确的 z-index 层级 */
  .splash-layer {
    z-index: 9999;  /* 开屏层 */
  }
  
  .input-bar {
    z-index: 100;  /* 输入区在 TabBar 上方 */
  }
  
  .zl-tabbar {
    z-index: 50;  /* TabBar 在最底层 */
  }
  ```
- **修正成本**：低（仅添加 z-index 数值）

## 4️⃣ 互动页 (page-prompt-interaction.md)

### 🟠 中优先级

#### 问题1：游戏网格的尺寸可能不合理
- **位置**：页面布局
- **问题**：提示词说"棋盘宽度用 `min(92vw, 420px)`"，但在很小的屏幕上（比如320px宽度的手机），92vw 是 294px，减去左右各16px的页面边距，剩余 262px。8×8网格，每个格子 262/8 ≈ 32px，小于44px触摸标准
- **影响**：在很小的屏幕上，格子太小，用户可能误触
- **建议**：
  ```css
  /* 减小格子数量（比如 6×6），或者增大最小宽度 */
  .game-board {
    max-width: min(92vw, 420px);
    min-width: 320px;  /* 保证每个格子至少 40px */
  }
  
  .game-cell {
    min-width: 40px;  /* 保证触摸友好 */
    min-height: 40px;
  }
  ```
- **修正成本**：中（可能需要修改游戏逻辑，支持 6×6 网格）

- **影响**：用户可能很快觉得"太简单"，失去挑战性

## 6️⃣ 设计规范 (design-system-spec.md)

### 🔴 高优先级

#### 问题1：部分颜色对比度可能不满足 WCAG AA 标准
- **位置**：颜色系统
- **问题**：品牌色 `--zl-brand: #87C8B4` 在白色背景上的对比度是多少？是否满足 WCAG AA 标准（4.5:1）？
- **影响**：如果对比度不足，视力障碍用户可能无法看清文字
- **建议**：
  - 使用对比度检查工具（比如 WebAIM Contrast Checker）检查所有颜色组合
  - 如果对比度不足，调整颜色（比如加深品牌色）
- **修正成本**：低（仅修改颜色值）

#### 问题2：设计规范中没有定义危险色（Danger Color）
- **位置**：颜色系统
- **问题**：设计规范定义了品牌色、背景色、文字色、边框色，但没有定义危险色（比如用于删除按钮、错误提示的红色）
- **影响**：如果出现需要危险色的地方（比如删除确认弹窗），开发者可能不知道使用什么颜色
- **建议**：
  ```css
  /* 添加危险色 */
  :root {
    --zl-danger: #FF5252;  /* 红色 */
    --zl-danger-light: #FFEBEE;  /* 浅红色背景 */
  }
  ```
- **修正成本**：低（仅修改设计规范 + tokens.css）
