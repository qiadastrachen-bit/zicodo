## 一、项目背景

在本次开发前，宠物对话系统存在以下两个核心问题：

1. **DeepSeek API 未实际生效**：后端 `.env` 文件中 `DEEPSEEK_API_KEY` 环境变量实际写的是 `DASHSCOPE_API_KEY`（变量名错误），导致代码中 `process.env.DEEPSEEK_API_KEY` 始终为 `undefined`，AI 对话永远走兜底响应（固定问候语），用户体验如同"智能客服，其实是个复读机"。

2. **宠物性格字段格式不统一**：后端 Pet model 的 `personality` 默认值为 `'温柔,好奇'`（中文标签），但前端 PetSettingsPage 选项保存的是 `'gentle'` / `'lively'` / `'tsundere'` / `'calm'`（英文标识符），导致：
   - 设置页刷新后高亮丢失（用户以为"没保存成功"）
   - AI system prompt 中 `性格标签：gentle` 对大模型意义不明确，无法体现性格
   - 数据库脏数据混杂中英文

本次开发目标：**接入真实 DeepSeek API，统一 personality 字段为中文标签，确保宠物对话与性格设置功能端到端可用。**

## 二、实施步骤

### 2.1 架构分析

项目中存在两套对话接口，均需支持中文 personality 和 DeepSeek API：

| 接口路径 | 实现文件 | 调用方 | 输出格式 |
|---------|---------|--------|---------|
| `POST /ziling/api/chat` | `src/routes/ziling.js` | 前端首页 PetHomePage.vue | JSON（quickReply + megachar + stream） |
| `POST /api/ai/chat` | `src/controllers/aiController.js` | 前端 api/ai.js（首页未使用） | SSE 流式文本 |

两套接口都通过 `process.env.DEEPSEEK_API_KEY` 判断是否调用真实大模型。

### 2.2 关键节点

**节点 1：环境变量修复（根因问题）**
- 发现 `.env` 中变量名为 `DASHSCOPE_API_KEY`，与代码中 `DEEPSEEK_API_KEY` 不一致
- 修正为正确变量名，填入真实 key `sk-e0188e88db23417281ecc294a9839950`
- `.gitignore` 已包含 `.env`，确保 key 不会提交到版本控制

**节点 2：personality 字段统一（前后端对齐）**
- 前端 stores/pet.js：`petPersonality` getter 默认值 `'gentle'` → `'温柔'`
- 前端 PetSettingsPage.vue：4 个选项 value 从英文标识符改为中文标签；默认选中值 `ref('温柔')`
- 后端 routes/ziling.js：增加 `toChinesePersona()` 兼容函数，自动把历史英文值转中文
- 后端 aiController.js：`buildSystemPrompt()` 增加英文→中文映射，确保 system prompt 永远是中文性格

**节点 3：对话接口鲁棒性增强**
- `/ziling/api/chat`：如果前端请求中没传 `persona`，从数据库 `pet.personality` 中取作为 fallback
- `/ziling/api/chat`：system prompt 中先 `persona = toChinesePersona(persona)` 再拼接
- `/api/ai/chat`：`buildSystemPrompt()` 中 `zhPersonality` 兜底为 `'温柔'`

**节点 4：测试验证（端到端）**
- 后端语法检查：`node --check` 通过
- 后端健康检查：`GET /api/health` → 200

- AI 对话（温柔性格）：`POST /ziling/api/chat` 带 `persona: '温柔'` + Authorization header → 200，返回体现"温柔"的回复
- AI 对话（活泼性格）：`POST /ziling/api/chat` 带 `persona: '活泼'` → 200，返回体现"活泼"的回复
- 前端构建：`npm run build` → 1962 模块，5.1s，无错误
- 前端开发服务器：`npm run dev` → http://localhost:5173/ 正常启动

### 5.3 AI 性格区分度观察

从多轮测试中观察到：
- **温柔性格**：回复偏向撒娇、安抚、柔弱语气（例：`主人喂我嘛～` `T_T` `Q_Q`）
- **活泼性格**：回复偏向热情、调皮、主动邀请（例：`转圈圈，嘻嘻~` `快来试试看吧！` `≥▽≤`）
- **改进建议**：部分回复对用户问题的针对性仍不足（例如问"不想写作业"却回复通用问候）。可在 system prompt 中增加**更严格的「必须直接呼应用户问题」约束**和**更具体的性格行为描述**来强化区分度。

## 七、后续改进建议

1. **加强性格区分度**：在 system prompt 中增加更具体的性格行为描述（例："温柔=使用柔和语气、安抚情绪、慢节奏；活泼=使用感叹号、积极鼓励、带动情绪"），并加入「必须直接回应用户问题」约束。
2. **增加对话上下文长度**：当前 `/ziling/api/chat` 传了 `history`，但 deepseek 请求时应把历史对话作为多轮 context 传入（当前 `callLLM()` 已支持 history 参数，前端已传，可测试是否需要更多历史记录）。
3. **增加 token 使用量日志**：监控 DeepSeek API 调用量和 token 消耗，防止超额度。
4. **对话缓存策略**：对常见问候类问题做本地缓存（如"你好""在吗"），减少 API 调用，降低成本。
5. **Pet model 增加 personality 约束**：在 migration 或 model 中对 personality 字段做枚举校验，防止新的脏数据写入。
