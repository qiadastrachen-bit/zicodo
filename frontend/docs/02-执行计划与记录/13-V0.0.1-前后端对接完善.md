## 🎯 一、改进目标

本次改进 **不进行功能创新**，仅聚焦于完善现有前端功能与后端 API 的对接，确保：
1. 用户在前端创建的任务能真正写入后端数据库
2. 用户打卡的任务能正确触发后端积分奖励 + 宠物经验升级
3. 前端展示的宠物信息与后端存储状态一致
4. 新增的 feedPet（喂养宠物）接口前后端打通

---

## 🔍 二、问题识别（对接缺口清单）

| # | 模块 | 问题描述 | 影响范围 |
|---|------|---------|---------|
| 1 | 任务 CRUD | `schedule.js` 中 addTask/updateTask/deleteTask 仅写 localStorage，未调后端 API | 用户创建的任务仅在本地可见，换设备/清缓存后丢失 |
| 2 | 任务打卡 | `toggleComplete` 仅切换本地 completed 状态，未调后端 `POST /tasks/:id/check` | 打卡无积分奖励，宠物经验不增长 |
| 3 | 任务初始同步 | 页面首次加载时未从后端拉取已有任务 | 与后端数据不一致 |
| 4 | 宠物喂养 | 前端 `petApi.feedPet()` 已定义，但后端 `routes/pets.js` 未注册 feedPet 路由 | 调用返回 404 |
| 5 | 字段映射 | 前端 schedule 使用 `date/recurrence/completed`，后端 Task 使用 `dueDate/repeatType/status+checkedToday` | 直接传递数据会导致字段不匹配 |

## 📝 三、具体修改计划

### 3.1 后端改动（1 项）

**文件**: `ziling-backend/src/controllers/petsController.js` + `ziling-backend/src/routes/pets.js`
- 新增 `feedPet(req, res)` 方法：给宠物 +10 经验，更新 mood=happy，检查升级
- 注册路由 `POST /api/pets/mine/feed`
- 返回：`{ pet, levelUp, expGain }`

### 3.2 前端 Store 改动（1 项）

**文件**: `frontend/src/stores/schedule.js`
- 新增 `toBackendPayload(task)`：前端字段 → 后端字段映射
- 新增 `fromBackendResponse(backendTask, existingTask)`：后端字段 → 前端字段映射
- 改造 `addTask`：先调 `taskApi.createTask`，成功后用返回数据更新本地状态；失败回退本地
- 改造 `updateTask`：先调 `taskApi.updateTask`，成功后更新本地；失败保留本地改动
- 改造 `deleteTask`：先调 `taskApi.deleteTask`，成功后清理本地；失败保留本地
- 改造 `toggleComplete`：调 `taskApi.checkTask`，返回 `{ task, pointsEarned, levelUp, pet }` 供上层使用
- 新增 `fetchTasksFromBackend()`：`onMounted` 时从后端拉取任务并合并本地闹钟字段

### 3.3 前端页面改动（2 个页面）

**文件**: `frontend/src/pages/tasks/CalendarPage.vue`
- 新增 `onMounted` 调用 `store.fetchTasksFromBackend()`
- 新增 `onToggleTask(taskId)` 包装函数：调 store.toggleComplete，成功后弹 Toast（积分/升级提示）
- 新增 `usePetStore` 引用：打卡成功后通过 `petStore.updatePetLocal(result.pet)` 同步宠物状态
- 插入 `ZlToast` 组件

**文件**: `frontend/src/pages/tasks/TaskCreatePage.vue`
- 同 CalendarPage：onMounted 同步、onToggleTask 包装、ZlToast 插入

## 🏗️ 四、数据流（新增后）

```
用户在日历页点击打卡
       ↓
onToggleTask(taskId)
       ↓
scheduleStore.toggleComplete(taskId)
       └→ POST /api/tasks/:id/check
       └→ 后端：奖励积分 + 更新宠物经验 + 升级检测
       └→ 返回 { task, pointsEarned, levelUp, pet }
       ↓
本地状态更新（tasks 数组）
       ↓
if (result.levelUp):
    Toast: "🎉 打卡成功！宠物升级到 LvX！+Y 积分"
    petStore.updatePetLocal(result.pet)  ← 首页宠物信息同步更新
else if (result.pointsEarned):
    Toast: "✅ 打卡成功！+Y 积分"
    petStore.updatePetLocal(result.pet)
```

## 🎯 八、架构设计原则重申

本次改进严格遵循以下原则（与现有架构一致）：

1. **数据流方向**：用户操作 → Store action → 后端 API → 更新本地状态 → 视图自动响应式更新
2. **优雅降级**：后端不可达时，保持本地功能可用，不阻塞用户
3. **模块化**：通用组件（ZlToast）保持独立，页面按需组合
4. **状态管理**：Pinia Store 为单一数据源，组件不直接持有业务数据
5. **字段适配层**：前后端字段不一致时，在 Store 层做双向映射，不污染 API 层和组件层
6. **命名规范**：沿用现有 `useScheduleStore` / `usePetStore` / `useTeamsStore` 命名风格

### 9.2 字段映射表（实际实现）

**前端字段 → 后端字段**

| 前端 schedule.js | 后端 Task | 映射方式 |
|-----------|-----------|---------|
| `title` | `title` | 直传 |
| `date`（YYYY-MM-DD` | `dueDate` | 重命名 |
| `startTime` / `endTime` | 无（本地仅前端保存为 note 字段） | 前端本地 |
| `note` | `description` | 重命名 |
| `recurrence` | `repeatType` | 字符串映射（`'none`→`none`, `'daily'`→`daily`, `'weekly'`→`weekly` |
| `completed` | `status` + `checkedToday` | 布尔 → `'completed'`/`'pending'` + `checkedToday: true` |

**后端字段 → 前端字段（反方向同理，`fromBackendResponse()` 做反向映射

### 9.3 错误容错策略（实际实现）
```
├─ addTask / updateTask / deleteTask：
   try { 调后端API } catch { 回退到本地操作 + console.debug }
   保证：即使后端挂了，用户本地操作仍能完成

toggleComplete：
   try { taskApi.checkTask } catch { 本地切换 completed }
```
