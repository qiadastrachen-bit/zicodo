### 3.1 问题现象
| 错误类型 | 错误消息 | 来源位置 |
|---------|---------|---------|
| HTTP 401 | 未授权，请重新登录 | [request.js:58](file:///d:/Ziling-app/backend/frontend/src/api/request.js#L58) |
| HTTP 400 | 用户名、邮箱和密码不能为空 | src.bak 旧代码 |

### 3.2 根本原因
- **直接原因**：后端可能正在运行或引用 `src.bak/` 目录的旧代码
- **间接原因**：备份目录与源代码目录同层级，容易混淆
- **旧代码问题**：旧版本要求同时输入用户名、邮箱、密码三个字段

### 3.3 当前代码验证（正确实现）

#### 注册接口验证
**文件**：[authController.js:12-50](file:///d:/Ziling-app/backend/src/controllers/authController.js#L12-L50)

```javascript
async function register(req, res) {
  const { username, email, password, nickname } = req.body;
  
  // 只验证用户名和密码
  if (!username || !password) {
    return fail(res, '用户名和密码不能为空');
  }
  
  // 邮箱自动生成
  const userEmail = email || `${username.slice(0, 50)}@temp.ziling.local`;
  
  // 创建用户
  const user = await User.create({
    username,
    email: userEmail,  // 使用自动生成的邮箱
    passwordHash,
    nickname: nickname || username
  });
}
```

**验证点**：
- ✅ 只要求 username + password
- ✅ 邮箱自动生成，格式：`{username}@temp.ziling.local`
- ✅ nickname 可选，默认为 username

#### 登录接口验证
**文件**：[authController.js:56-93](file:///d:/Ziling-app/backend/src/controllers/authController.js#L56-L93)

```javascript
async function login(req, res) {
  const { email, password } = req.body;
  if (!email || !password) return fail(res, '昵称和密码不能为空');
  
  // 支持邮箱或用户名登录
  let user = await User.findOne({ where: { email } });
  if (!user) {
    user = await User.findOne({ where: { username: email } });
  }
  
  // 体验服模式：不存在自动创建
  if (!user) {
    const autoEmail = `${crypto.randomUUID()}@ziling.local`;
    user = await User.create({
      username: email,
      email: autoEmail,
      passwordHash,
      nickname: email
    });
  }
}
```

**验证点**：
- ✅ 支持邮箱或用户名登录（参数名仍为 email）
- ✅ 体验服模式自动创建用户
- ✅ 自动生成唯一邮箱

## 五、经验教训总结

### 5.1 开发规范改进建议
1. **备份目录管理**
   - 备份目录不应放在项目根目录
   - 建议移至 `backup/` 或 `archive/` 子目录
   - 备份完成后及时移开或删除

2. **服务启动检查**
   - 启动前确认工作目录
   - 检查是否有旧代码残留
   - 使用绝对路径引用模块

3. **错误日志记录**
   - 记录错误来源文件路径
   - 记录代码版本号或 git commit
   - 便于快速定位问题

### 5.2 调试流程改进
1. **系统排查流程**
   - 先搜索错误消息的精确来源
   - 检查是否有备份或临时文件
   - 确认正在运行的代码版本

2. **验证方法**
   - 对比当前代码与备份代码
   - 检查 package.json 启动脚本
   - 验证进程工作目录
